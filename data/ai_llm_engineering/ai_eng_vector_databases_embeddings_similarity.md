### Vector Databases, Embeddings & Similarity Search

**What embeddings are:**
- Dense numerical vectors representing the meaning of text/images
- Similar content → similar vectors (close in vector space)
- Generated by models: OpenAI text-embedding-3-small, Cohere embed, Sentence-BERT

**Embedding dimensions:**
- text-embedding-3-small: 1536 dimensions
- text-embedding-3-large: 3072 dimensions
- Smaller models: 384-768 dimensions
- Tradeoff: more dimensions = more accuracy + more storage/compute

**Similarity metrics:**
| Metric | Formula | When to use |
|--------|---------|-------------|
| Cosine similarity | dot(A,B) / (‖A‖·‖B‖) | Normalized embeddings (most common) |
| Euclidean (L2) | √Σ(ai-bi)² | When magnitude matters |
| Dot product | Σ(ai·bi) | When vectors are already normalized |

**Vector databases:**
| Database | Type | Best for |
|----------|------|----------|
| **Pinecone** | Managed SaaS | Easiest, fully managed |
| **pgvector** | PostgreSQL extension | Already using PostgreSQL, moderate scale |
| **Weaviate** | Open-source, self-hosted | Hybrid search, rich features |
| **Qdrant** | Open-source, Rust | High performance, filtering |
| **Chroma** | Open-source, lightweight | Prototyping, development |
| **Milvus** | Open-source, distributed | Large scale, production |
| **FAISS** | Library (not a DB) | In-memory similarity search |

**pgvector (PostgreSQL):**
```sql
CREATE EXTENSION vector;

CREATE TABLE documents (
    id SERIAL PRIMARY KEY,
    content TEXT,
    embedding vector(1536),  -- 1536 dimensions
    metadata JSONB
);

-- Create index for fast similarity search
CREATE INDEX ON documents USING ivfflat (embedding vector_cosine_ops) WITH (lists = 100);
-- or HNSW index (better accuracy, more memory):
CREATE INDEX ON documents USING hnsw (embedding vector_cosine_ops);

-- Query: find 5 most similar documents
SELECT content, 1 - (embedding <=> query_vector) AS similarity
FROM documents
ORDER BY embedding <=> query_vector  -- <=> is cosine distance
LIMIT 5;
```

**Index types for approximate nearest neighbor (ANN):**
| Index | How | Tradeoff |
|-------|-----|----------|
| Flat/brute force | Compare with every vector | Exact, slow at scale |
| IVF (Inverted File) | Cluster vectors, search nearest clusters | Fast, some recall loss |
| HNSW (Hierarchical Navigable Small World) | Graph-based navigation | Best recall, more memory |
| PQ (Product Quantization) | Compress vectors | Low memory, lower accuracy |

**Quantization (reduce storage/cost):**
- Binary quantization: 1536 floats → 1536 bits (32x compression, some quality loss)
- Scalar quantization: float32 → int8 (4x compression, minimal quality loss)
- Matryoshka embeddings: truncate dimensions (1536 → 256) for coarse search

**Rule of thumb:** pgvector if you're already on PostgreSQL and have < 10M vectors. Pinecone for managed production. HNSW index for best recall. Start with cosine similarity. Quantize for cost at scale. The embedding model choice matters more than the vector DB choice.
