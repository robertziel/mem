### Tokenization (VTS) — PAN Replacement and Cryptogram Verification

**Visa context:** Visa Token Service (VTS) replaces the real card number (PAN) with a surrogate token (DPAN) for each merchant/wallet. If a merchant is breached, tokens are useless without Visa's de-tokenization service. Each transaction includes a cryptogram proving the token was used legitimately.

**How it works:**
```
Provisioning (one-time):
  Cardholder adds card to Apple Pay/Google Pay/merchant vault
  1. Token requestor sends PAN to Visa Token Service
  2. Visa verifies card with issuer → issuer approves
  3. VTS generates token (DPAN) + unique derived key (UDK)
  4. Token stored in wallet/merchant system — PAN is NOT stored

Transaction:
  1. Wallet/merchant submits token (DPAN) + cryptogram (AC)
  2. Acquirer → VisaNet
  3. VTS de-tokenizes: token → real PAN
  4. VTS verifies cryptogram (proves legitimate use, not stolen token)
  5. VisaNet routes to issuer with real PAN
  6. Issuer authorizes normally
  7. Response flows back with token (not PAN)

  Merchant never sees, stores, or transmits the real PAN.
```

**Token types and domain restriction:**
```
Token scope (domain control):
  - E-commerce token: bound to specific merchant
  - Device token: bound to specific device (Apple Pay, Google Pay)
  - Card-on-file token: bound to merchant + recurring billing

Domain control:
  Token for Merchant A cannot be used at Merchant B
  Token for Device X cannot be used on Device Y
  Cryptogram proves the token is being used in its intended domain

If token is stolen:
  - Useless without the cryptogram
  - Useless outside its domain
  - Can be revoked without reissuing the physical card
```

**Cryptogram (Application Cryptogram — AC):**
```
Generated by: wallet/SDK using Unique Derived Key (UDK)
Input: token + transaction amount + timestamp + device data + counter
Output: one-time cryptogram (changes per transaction)

Verification: VTS verifies cryptogram using the same UDK
  If valid → token is legitimate, proceed
  If invalid → reject (stolen token or replay attack)

This is why tokenized transactions are more secure than raw PAN:
  - PAN is static (same for every transaction)
  - Token + cryptogram is dynamic (unique per transaction)
```

**Token lifecycle management:**
```
Events:
  Card reissued (new expiry) → VTS updates all tokens automatically
  Card replaced (new PAN)    → VTS provisions new tokens, deactivates old
  Card reported stolen       → VTS suspends all tokens
  Merchant compromised       → revoke merchant's tokens, card stays active

Key benefit: cardholder doesn't need to update payment info everywhere
  VTS pushes updates to all token requestors automatically
```

**General software pattern: Token Vault / Proxy Encryption**

```ruby
# Same pattern: tokenize sensitive data in your application
class TokenVault
  # Store real value, return token
  def tokenize(pan)
    token = generate_token(pan)
    encrypted_pan = encrypt(pan)

    VaultEntry.create!(
      token: token,
      encrypted_value: encrypted_pan,
      domain: request_domain  # restrict where token can be used
    )

    token  # return token, never the PAN
  end

  # Resolve token to real value (internal use only)
  def detokenize(token, domain:)
    entry = VaultEntry.find_by!(token: token)
    raise DomainViolation unless entry.domain == domain
    decrypt(entry.encrypted_value)
  end

  private

  def generate_token(pan)
    # Preserve format: same length, same BIN prefix, different number
    # This way existing systems expecting a card number still work
    prefix = pan[0..5]  # keep BIN (identifies card network/issuer)
    suffix = SecureRandom.hex(5)
    check = luhn_check_digit("#{prefix}#{suffix}")
    "#{prefix}#{suffix}#{check}"
  end
end

# Usage: merchant stores token, never the PAN
class PaymentController < ApplicationController
  def create
    token = TokenVault.tokenize(params[:card_number])
    order.update!(payment_token: token)  # safe to store
    # To charge: detokenize internally, send PAN to processor
  end
end
```

**Where you see this pattern:**
- Stripe: `tok_xxx` replaces card numbers, `pm_xxx` for payment methods
- AWS KMS: encrypt data keys, store encrypted version, decrypt on demand
- OAuth: access tokens replace user credentials
- Database: PII vault (store encrypted PII separately, reference by ID)
- Session tokens: replace user credentials for subsequent requests

**Rule of thumb:** Never store the real sensitive value — store a token that maps back to it through a controlled service. Bind tokens to a domain (which merchant, which device) so stolen tokens are useless. Include a per-transaction cryptogram for dynamic proof of legitimacy. Manage the token lifecycle centrally — when the underlying credential changes, update all tokens without disrupting users.
